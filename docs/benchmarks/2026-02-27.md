## 2. Lazy-Load Comparison

Overlay loading strategy impact on startup time (10 runs each):

| Strategy | Average | Stddev | Notes |
|----------|---------|--------|-------|
| Current (lazy overlay) | 477.6ms | 42.6ms | Overlays loaded on first call |
| Eager (taskwarrior only) | 435.6ms | 7.9ms | `overlay use` at startup |
| Eager (all overlays) | 476.7ms | 33.1ms | taskwarrior + lima at startup |

**Delta (lazy vs eager taskwarrior)**: -41.9ms (eager is faster)

**Observation**: The lazy approach has higher variance (42.6ms vs 7.9ms stddev).
The average difference is within noise for all-overlays case (-0.9ms).
Lazy-load does not provide measurable startup time benefit.

## Summary & Implications for Refactoring

### Phase 1 (LaunchAgent): No performance impact expected.

### Phase 2 (Python scripts): No performance impact expected.

### Phase 3 (integrations):
- The hash-based staleness check adds ~358ms per startup (4 shasum calls)
- Nix derivation approach (Plan B) would eliminate this entirely since
  init scripts would be generated at `darwin-rebuild switch` time
- Trade-off: Plan B ties init script regen to `darwin-rebuild switch` instead
  of detecting binary changes automatically

### Phase 5 (lazy-load):
- **Decision: lazy-load should be removed.**
- Startup time difference between lazy and eager is <100ms threshold
  (actually -41.9ms for taskwarrior, -0.9ms for all overlays)
- Lazy approach adds code complexity (overlay wrappers in every function)
  with no measurable benefit
- Eager approach has lower variance (more consistent startup time)
